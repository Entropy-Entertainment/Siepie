# Workflow within Siepie and Takkie

This document has all the miscellaneous information that does not fit in other documentation files.

## Planning
For planning, we use Jira. When creating a new feature do this in the agile form.
Userstories are written in the story issue type. If the usertory discribes a bug do this in the bug issue type.
Epics are used to categorize userstories. Tasks are used for smaller tasks that do not require a full userstory. A DoD required for tasks, bugs and userstories alike.


## Issue properties
We work with the priority and story points properties of issues. Priority is determined by the product owner. Story points are determined by the team during refinement sessions. 

## Issues mid sprint
Issues can be added to the sprint during the sprint. If approved by the product owner, the worklow in this doucment is followed. Refinement can be done over online channels in this senario and needs only half of the people.

## Finishing an issue
An issue is finished when it meets it's the definition of done. Every feature has it's own tests that check the DoD (and if thought necessary more). If the tests pass a pull request can be made and the product owner will review the pull request. Once passed it's merged with development and moved to done.

## Not finishing an issue

If an issue is not finished in a sprint it is moved to the new sprint.
